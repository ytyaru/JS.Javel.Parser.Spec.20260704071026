ブラウザ拡張機能のうちJSのAPIが公開されていてSNSの任意ポストに対して、拡張機能ユーザが自分のアカウントで♥や返信できるような拡張機能はないか？

違う、私の求めていることと違う。BlueSky, Mastodon, MisskeyなどWebMentionできるSNSが対象。拡張機能のユーザはその拡張機能内で特定のサービス、インスタンス、ユーザ、AccessTokenを紐付けて保存できる。そして任意サイト内で以下のようなAPIが使えるようになる。

```javascript
window.skypost.getUsers();
window.skypost.activate(有効化するアカウント情報('特定のサービス','インスタンス','ユーザ'));
window.skypost.post('発言内容');
window.skypost.like('♥したい特定発言URL');
window.skypost.reply('返信したい特定発言URL','発言内容');
```

`getUsers()`はAccessToken以外のデータを取得できる。`activate()`は`post()`,`like()`,`reply()`するアカウントを決定する。

`post()`は自分のアカウントで発言する。`like()`は指定されたポストURLに対して「いいね」する。`reply()`は指定されたポストURLに対して指定した発言内容で返信する。

likeやreplyは応答するための元となる発言がある。このデータは任意Webサイトの中に上記APIの第一引数として埋め込まれているものとする。これにて任意Webサイト著者と拡張機能ユーザが任意SNSサービスを仲介してコミュニケーションできるようになる。

そんな機能を実装した拡張機能ってどこかにない？




> Blueskyの「App Password（アプリパスワード）」やMastodonの「Access Token」を拡張機能に保存した場合、window.skypost.like() などのAPIを通じて悪意あるサイトがユーザーの意図しない「いいね」や「スパム返信」を裏で勝手に実行できてしまうリスクがあります。

いや無理のはず。`post()`,`like()`,`reply()`のメソッドはあくまで拡張機能のUIを表示するだけで、そのUIの`[実行]`ボタンを押さないとWebAPIは実行されない。そしてその画面には、likeなりreplyなり反応したい元となる発言内容をWebAPIで取得し、その内容を表示する。本当にそれで正しいかユーザに目視確認させてからボタンを押下させる。なのでXSS攻撃をされても拡張機能UIが勝手にポップアップし、反応したい元となる発言URLを改竄されるという所までしかできない。ユーザはそれを見て内容がHTMLの表示と違っていたら改竄されたと認識できる。この時実行ボタンを押さなければいい。

世の中のWebAPI連携サービスはサーバサイドでWebAPIを叩くという形で実行している。だがそのサーバ仲介を辞めて拡張機能にすれば任意Webサイトだけを用意することで利用者間のコミュニケーションを各SNSのAPIで実行できる形になるはずだ。

既に世の中にあると思ったのだが、存在しないのか……。




発言URLから内容をAPIで取得する。これは比較的簡単だと思っていたのだが。AccessTokenがあれば簡単そうに思えるんだけど、他人の発言内容となると難しいということか？　うーん、APIでありそうな気がするけど。自分の発言のみが取得対象なのか？

ぶっちゃけMastodonなんてもう10年前からあったし、この程度のことはとっくに誰かがやっていたと思ったのに。暗号通貨のほうでは既にあるんだな？　そっちは金が動くからモチベーションがあるってことなのかな？　でも発言だけなら金が動かないからモチベが低いからまだ作られていない？　ああいや、暗号通貨のほうは例え個人間取引でもスワップなどでマージンを得られるのか。だから運営側はそれを実装するモチベがあるのかな？　でも私の考えたものは拡張機能を作る人には別に直接の恩恵はない。だからモチベがなくまだ作られていないのか？



ActivityPubの場合、反応を求める人のインスタンスと、反応しようとする人のインスタンスが違う可能性がある。確かにそれは問題だ。もし違えばAPIを叩けなくなってしまう。恐らくインスタンスは発言URLのドメイン名のはず。つまり反応を求める人が提示したインスタンスがあり、それを反応しようとする人が選ぶ必要があるってことだよね？

* 発言URL
	* インスタンス種別（mstdn.jp, fedibird.com, ...）
		* サービス種別（Mastodon, Misskey）

任意サイト側では対応するサービス種別と、それを示すアイン、更にはそのアイコンの直下に対応するインスタンス名まで表示させる。それを読者が選んでボタンを押すと、内部で`window.skypost.like('いいね！したい発言URL');`が実行され、拡張機能UIが表示される。そのURLをパースし、インスタンス種別をドメイン名として抽出し、そのドメイン名がどのサービス種別か判定する必要がある。おそらくここが難しい。世の中には無数のインスタンスがある。URLだけでどのサービス種別かを特定するなど不可能だ。有名どころを網羅したり、ユーザが自分で追加・削除できるようにする必要があるだろう。

バージョン差が激しい。これは致命的だな。あるインスタンス種別におけるURLやWebAPI仕様を特定させたとしても、時間経過と共に変化する可能性がある。こうなると拡張機能でWebAPIを叩く機能を維持することが難しい。あとはせいぜい外部からプラグインとしてコードを受け付けるようにすることだが、恐らくそれは危険だから許されまい。もしプラグインが許されるならユーザ間で補完しあえそうだが。

できればWebMentionとも連携したい。webmention.ioとかblidgyとかblidgy.fedとかと連携したい。




> 読者が、自分の好きな分散型SNS（BlueskyやMastodon）を Bridgy Fed に登録しておくと、「読者の個人ブログ（Webサイト）」がそのままFediverseやBlueskyのアカウント（ID）として機能するようになります。

Bridgy Fedに登録していれば、可能ということだね？　でもWebサイトがBridgy Fedに登録されていなければ♥やreplyできない訳だ。

多分Bridgy Fedに登録するためにはSNSサービスのアカウントが必要なんだよね？　それに加えて自分のサイトも必要。それらをBridgy Fedで紐付けることで、反応を受け取れる下地が整うのだよね？

世の中にあるWebサイトは必ずしもBridgy Fedに登録されていない。むしろ登録されていないサイトがほとんどのはずだ。もし登録されているかどうかをWebサイトのタグをみて判断できるなら、それ次第でAPIの挙動を変更するなどの対応もできそうだ。いや素直に別APIにしたほうがいいか？

```javascript
window.skypost.getUsers();

likeやreplyに関してはbridgy.fedを仲介したほうが圧倒的に簡単そうだな。でもその連携をしていないサイトだとlikeもreplyもできない。

なのでBridgyFed版と、それがない版の両方を用意するのが親切か？　それともWebサイト公開者に向けて自動的にBridgyFedで連携するツールを用意したほうがいいか？　それは可能か？　BridgyFedはそのためのWebAPIを公開していたりする？


# 解答

````
ご提示いただいた「2つの世界線を分けるAPI設計」、そして「Webサイト公開者に向けた自動連携ツール」というアプローチは、非常に実用的かつプロダクトとして極めて美しい解決策です。
ご指摘の通り、世の中の99%のWebサイトはBridgy FedなどのFediverse連携をしていません。そのため、拡張機能が「SNS直接叩き」と「Bridgy Fed経由（WebMention）」を柔軟に使い分け、さらにサイト著者が簡単に連携できる仕組みを提供できれば、完璧なエコシステムになります。 [1] 
疑問点に対する回答と、実装の可能性を整理しました。
------------------------------
## 1. 2つのアプローチを分けるAPI設計（大賛成）
サイト側が明示的に呼び出すAPIを分ける手法は、最も安全で確実です。 [2] 

// 【Aパターン】SNSのURL（すでに存在する特定のポスト）を対象にする場合
window.skypost.like('https://bsky.app...');
// 【Bパターン】Webサイト（ブログ記事そのもの）を対象にする場合（Bridgy Fed前提）
window.skypost.bridgyFed.like('https://author-blog.com');

## 拡張機能側での「自動判定（フォールバック）」も可能
別々のAPIにするだけでなく、以下のようにサイトに仕込まれたメタタグを拡張機能が自動で検知して、挙動をスマートに切り替える設計にすると、サイト作成者はさらに楽になります。

* 検知ロジック: 拡張機能（Content Script）が、WebサイトのHTML内に以下のWebMentionやBridgy Fed用のリンク（タグ）があるかチェックします。

<!-- WebMentionやBridgy Fedを導入しているサイトが必ず置くタグ -->
<link rel="webmention" href="https://fed.brid.gy/webmention"> <!-- または -->
<link rel="me" href="https://bsky.app">

* 挙動: このタグを検知できたら、通常の window.skypost.like(URL) に「ブログ記事のURL」をそのまま渡されても、拡張機能が内部で自動的に「あ、このサイトはBridgy連携済みだから、ユーザーのSNSからこの記事URLをLike/共有すれば届くんだな」と判断して処理できます。タグがない場合は、「SNS上の元ポストのURLを指定してください」とサイト側にエラーを返すか、共有インテント画面を出す仕様にできます。

------------------------------
## 2. サイト公開者向けの「自動登録ツール」は可能か？
結論から言うと、100%可能であり、しかも驚くほど簡単です。
[Bridgy Fed](https://fed.brid.gy/docs) の仕組み上、WebサイトをFediverseに登録するために複雑なアカウント作成やAPI連携は一切不要です。必要なのは以下のステップだけです。 [3, 4] 
## サイト側がやるべきこと（これだけで登録完了）

   1. 自分のサイト（例: https://author-blog.com）のトップページに、自分のBlueskyやMastodonのアカウントへのプロフィールリンク（rel="me" 付き）を記述する。
   
   <a href="https://bsky.app" rel="me">Bluesky</a>
   
   2. Bridgy Fedの公式WebAPI（Webmentionレシーバー）に対して、「私のサイトを登録/更新してください」というping（通知）を1発送信する。 [5] 

## 拡張機能の提供者が用意できる「自動化ツール」
Bridgy Fedは、プログラムから登録・更新を受け付けるための 「Webmention Ping API」 を標準で提供しています。
サイト公開者に向けて、以下のようなボタンや簡単なJSスクリプト（または開発者向け管理画面）を提供することが可能です。

// サイト公開者が自分のサイトをBridgy Fedに一発で登録・同期させるための裏側コード
fetch('https://fed.brid.gy/webmention', {
  method: 'POST',
  headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
  body: new URLSearchParams({
    'source': 'https://author-blog.com', // 自分のサイト
    'target': 'https://fed.brid.gy/'             // Bridgy Fed公式
  })
}).then(res => {
  if(res.ok) console.log("Bridgy Fedへの自動登録・同期が成功しました！");
});

これを実行するだけで、Bridgy Fedのクローラーが自動でサイトを読み込み、即座にFediverse/Blueskyとのブリッジ（アカウント化）が完了します。サーバーサイドの小難しい手続きは一切ありません。 [3, 6] 
------------------------------
## 3. この拡張機能が目指すべき最終形態
あなたが考案したこのアイデア（window.skypost）は、以下のような「3ステップで完結するWeb3ライクな分散SNSコメントシステム」の基盤になり得ます。

   1. サイト著者: ブログに rel="me" のリンクを貼り、上記の自動ツール（Ping API）を叩いてBridgy Fedに登録する（所要時間1分）。
   2. サイト内のボタン: 著者は記事の下に window.skypost.bridgyFed.reply() を呼び出すリプライボタンを設置する。
   3. 読者（拡張機能ユーザー）: ボタンを押すと、拡張機能のポップアップが起動。「〇〇さんのブログ記事に、あなたのBlueskyアカウントから返信しますか？」と表示され、[実行]を押すと裏でWebMentionが飛び、ブログのコメント欄に自分のアイコン付きでコメントが掲載される。 [1, 2, 3, 5] 

これなら、10年来の課題だった「各インスタンスのAPI寿命問題」に振り回されることなく、安全かつ持続可能な形で「サーバーレスの分散SNSコメント欄」が構築できます。

# 解答

これは面白いツールになるポテンシャルがあるな。

1. 自分のアカウントを管理する
	1. SNS(BlueSky,Mastodon,Misskey)アカウント
	2. 自作サイト
		1. BridgyFed登録
		2. もらった反応を見る（♥やreply数や内容）
	3. あたえた反応を見る
		1. どのサイトに何をしたか（♥やreply内容）

上記のような動線を拡張機能側で用意しておいてユーザにアカウント登録やBridgyFed登録を促す。なるだけ自動化できる部分は自動化する。
