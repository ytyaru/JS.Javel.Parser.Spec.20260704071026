`｜《《単語》》`や`｜《｜《単語｜》｜》`ではエスケープできないの？


ルビや傍点を振る独自構文を考えたい。構文はその構文自身をプレーンテキストで表示できるようエスケープできることまでをセットにして設計したい。

ルビはエスケープ以外はカクヨムの方法でいい。但しエスケープ方法については何かいい方法を考える必要がある。

傍点は`＊傍点＊`か`﹅傍点﹅`がいいと思う。同じエスケープ方法を考えたい。

インライン構文を同一のエスケープ方法にするならば半角バッククォートで囲むという案があるが。できれば日本語入力である全角文字ですべて完結できるのが望ましい。

提案ある？



エスケープ化するために`＼`というメタ文字を用意するわけか。確かにメタ文字の数が少ない時はこれが一番楽そうだ。メタ文字の字種が変わっても不変の法則なので覚えやすいのも良い。ただ、メタ文字の数が増えると面倒になる。パース処理も複雑化しそう。

囲み文字で囲う案。`｛｝`でエスケープするわけか。確かにこの文字なら素早く入力できて良い。`＼｜漢字＼《かんじ＼》`のように3字適切な箇所にエスケープコードを入力するのが面倒な場合`｛｜漢字《かんじ》｝`のように両端の`｛｝`さえ外せばいつでもメタ文字として解釈できるのも楽でいい。ただ、`伏字＼＊`のように一字だけエスケープしたい時には`伏字｛＊｝`と書かねばならないのがやや冗長か。

* エスケープはできないものとする
* エスケープ文字として`＼`を用意する
* エスケープ括弧として`｛｝`を用意する
* `＼`と`｛｝`を両方用意する
* 上記全パターンを選べるようにする

同じ記号を二つ重ねる形式は微妙だ。たとえば`##`や`《《》》`のようなメタ文字がそのメタ文字の数によってエスケープではない別の意味として解釈されるパターンがある。将来の拡張を考えると、メタ文字の字種によって重ねる形式が使えないケースも出てくると思われる。

エスケープ構文は`｛｝`だけを用意するのが楽であり統一性と汎用性のある書式になるだろう。

傍点のメタ文字`＊`だが、これはプレーンテキストで使う場面が比較的多いように思う。例えば伏字にして著作権に配慮するだとか、作中のシーンにおいて古い石版の文字が読み取れなかったような表現に使うとか。更には主題区切りや場面転換の時にも使われる。

かといって`﹅傍点﹅`のようにするのは入力が少々手間だ。`＊`には及ばない。`《《傍点》》`はカクヨムやいくつかのサイトで使われる記法だが冗長だ。メタ文字種の数を減らせるのは利点だが。他に何かいいメタ文字はないか？




中黒`・`は`＊`以上に汎用的に使うのでメタ文字にするのは危険だ。「プレーン・テキスト」のようにカタカナ英語において単語区切りとして頻出する。そのたびに`プレーン｛・｝テキスト`と記述せねばならず面倒極まりない。箇条書きで使われることもあるし。

全角ハット`＾`は恐らく使われないだろう。私が知っているのはせいぜい「心がぴょんぴょんするんじゃぁ〜＾」とか、声が裏返るようなことを表現する時に使われるのを見たことがあるくらいか。

全角不等号はない。それならせめて`＜傍点＞`だろう。でも私なら`＞取り消し線（または❌）＜`みたいな使い方をしたほうが視覚的に分かりやすいと思う。しかしそもそも全角不等号をメタ文字にはしないほうが良いと思う。`小＜中＜大`みたいなプレーンテキスト表記ができなくなってしまうのは厳しい。本格的な小説ではそんな記述しないとは思うけど。WEB小説では見たことがある。

`《傍点》`はない。カクヨム記法と同じにするならば`漢字《かんじ》`でルビになって欲しいから。でも`《》`という記号は傍点などの強調というニュアンス的に相応しいとは思う。だとしたらルビのほうが`｜親文字＜ルビ文字＞`のようにすべきなのでは？　と思う。でも大本である青空文庫がなぜルビ文字を`《》`にしたかといえば`＜＞`はプレーンテキストで割と使う文字だからだと思われる。もしエスケープ囲みの仕様が無ければ`｜漢字｛かんじ｝`でも良かったかもしれない。でんでんマークダウン記法に似ている。だがこうしてしまうとエスケープはどうするの？って話になる。

`＿傍点＿`というのは微妙だな。下線を引くメタ文字に見える。それと英語圏の斜体と日本語圏の傍点はニュアンスを匂わせるという点において共通だが、その表現方法が違う。日本語では斜体にすべきでない。傍点で匂わせるべきだ。

私は表紙に関しては以下のような構造にしたい。

* 最初の見出しが登場するより前が表紙である
* それ以降が本文である

表紙の構造は以下のようにしたい。なるだけメタ文字やキー名を書かずに済むように工夫する。

```
キャッチコピー
https://
...
タイトル
https://
...
著者名
https://
...
MONAコインアドレス

説明文(80字)。

帯文(200字)。

カテゴリ
ジャンル
タグ
注意
```

表紙のうち必須なものはタイトルと著者名のみ。それ以外はすべて任意。

表紙は4つの段落で構成される。一つの段落は2連続改行コードで区切る。

段落|項目
----|----
1|キャッチコピー、タイトル、著者名、コイン
2|説明文
3|帯文
4|カテゴリ、ジャンル、タグ、注意

必須項目はタイトルと著者名のみ。それ以外はすべて任意。URLもすべて任意。

URLは複数入力可能。理由は複数のストレージやSNSサービスが同一対象を示す場合があるから。キャッチコピーのURLは受賞した時などにその情報源をURLで示す。

説明文はhtmlのmeta description。Google検索時は先頭80字まで表示する。

帯文は本にある帯文を模したもの。200字まで。

カテゴリ・ジャンルは任意で大分類〜小分類などの包含関係を表現できるよう以下のように`＞`,`＜`をメタ文字にする。

```
大＞中＞小
小＜中＜大
```

カテゴリとジャンルは例えば以下のように使い分ける。書類としての分類と、内容としての分類に分ける。

```
創作文＞小説
ファンタジー＞ダーク
```

タグは全角スペース区切りとする。作品やその対象者を示すキーワード。

```
タグ１　タグ２　タグ３
```

注意は以下のいずれか。全角スペース区切り。読者に対してこのような描写があると注意する。

```
性　暴力　残酷
```

将来的にフェンス構文でYAML形式の設定データをフロントマターのようにセットする仕組みを作るかもしれない。これ以上複雑化したらキー名を割り当てざるを得ないだろう。流石に行位置だけで項目を特定するのはこのくらいの規模が限界だろうから。その時はこの構造の直下に記述することにする。



パースでバグらないか心配な所はキャッチコピーが任意である所だ。１段落目の何行目は何のデータか。これを特定することが確実にできると言えるロジックを論理的に組み立て可能であるかどうか。これを考えたい。

1. 原稿内容をすべて読む
2. 最初の見出しから前を表紙として読む
3. 表紙のうち一段落目を読む

ここまで終えたとして、あとは一段落目にあるであろうキャッチコピー、タイトル、著者名、MONAコインアドレス、URLたちをどう紐付けるか。どの位置にある文字列は、何を意味するものか。それを正確に定義できるか。

まずタイトルと著者名は必須だから絶対に２行はあるはずだ。２行より少なければエラーにすべきである。２行だけしかなければそれは一行目がタイトルで、二行目が著者名と確定する。たとえその内容が`https://`で始まっていようが？　いやそれは困る。おそらくそれは意図していない。よってキャッチコピー、タイトル、著者名、MONAコインアドレスの値は、`https://`で始まらない文字列であるべきだ。

URLは`https://`で始まるはずだから簡単に特定できる。

ややこしいのは必須でない任意の項目があることと、URLは一個だけでなくゼロから無限個まで好きなだけ入れることができてしまうことだ。これらが組み合わさって膨大なパターンがありえる。

## 1. 4段落目を「4行必須」とするパースの絶対ルール
4段落目は、内容（テキスト）に固有のパターン（https:// のようなもの）を持たないため、「行のインデックス（1〜4行目）」だけが唯一の信頼できる情報源になります。

* 1行目： カテゴリ（※空値ならデフォルト値「創作文＞小説」をシステムが適用）
* 2行目： ジャンル
* 3行目： タグ
* 4行目： 注意
